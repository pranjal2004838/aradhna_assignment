# Case Study — Aradhana Ritual Companion

## Overview

**Aradhana** is a mobile-first devotional web app that turns scattered prayers
into a steady daily practice. It was built as a portfolio MVP to demonstrate a
complete product — from a polished landing page, through authentication and
onboarding, to five working feature areas — using a small, readable codebase.

- **Type:** Web app (mobile-first, responsive)
- **Stack:** Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS v4,
  Framer Motion
- **Data:** Browser `localStorage` (no backend, no API keys)
- **Status:** Lint clean, production build passes, all core flows tested in-browser

---

## The problem

People who want a daily spiritual habit struggle with three things:

1. **Consistency** — there's no single place that nudges them and tracks a streak.
2. **Guidance** — they don't always know what to read or chant on a given day.
3. **Safety** — when they ask the internet sensitive questions (health, money,
   legal, "what will happen to me?"), they get unreliable or unsafe answers.

## The solution

A calm, single-purpose app with:

- a **personalized daily ritual** (Read · Chant · Reflect),
- a **jaap counter** with a mala ring and a 108 target that survives refresh,
- a **content library** of traditional texts with search and a clean reading mode,
- **Ask Guruji**, a deliberately *safe* reflection chat, and
- **Sankalp Path**, a guided 7-day journey based on the user's intention + deity.

---

## Key design decisions

### 1. Self-contained, no backend
All user data is stored in `localStorage`, namespaced by user id. This made the
app instantly demoable with zero setup and kept the code easy to explain. The
honest tradeoff: data is per-browser and passwords are not securely hashed, so
this is an MVP pattern, not a production auth system.

### 2. Safety-first "Ask Guruji"
Instead of calling a live AI, Guruji uses a small, readable rule engine in
`lib/guruji.ts`:

- **Unsafe categories** (medical, financial, legal, prediction, harm) are matched
  by simple patterns and always get a calm refusal that redirects to a qualified
  professional.
- **Friendly topics** (focus, calm, discipline, gratitude, sleep, anger) get warm,
  reflective answers.
- Everything else gets a gentle default reflection.
- A **10 questions/day** limit prevents abuse.

This means the assistant **cannot** produce dangerous advice — the safety is
structural, not hopeful.

### 3. Predefined Sankalp templates
A Sankalp is generated by combining an **intention spine** (7 daily titles +
reflections) with a **deity pack** (mantra + suggested reading). This produces
36 possible journeys (6 intentions × 6 deities) that are always coherent and
calm, without needing an AI.

### 4. One consistent design system
All colours, gradients, fonts and animations live in `globals.css` as theme
tokens ("Cosmic Bhakti" palette). Every screen pulls from the same tokens, so
the look stays consistent and is trivial to retheme.

## Architecture

```
UI screens (app/)  ──►  small helpers (lib/store.ts, lib/auth.tsx)  ──►  localStorage
        ▲                                   │
        └──────────  React context (AuthProvider) shares user + profile
```

- `AuthProvider` loads the session once and shares it via context.
- `AppShell` guards every private route (redirects to `/login` or `/onboarding`).
- Feature pages never touch `localStorage` directly — they call typed helper
  functions, which keeps storage keys and shapes in one place.

See `README.md` for the full file tree.

## Testing

All core flows were tested manually in the browser (Chrome). Highlights:

| Area | Result |
| --- | --- |
| Signup + email/password validation | Pass — invalid email and mismatched passwords are blocked |
| Onboarding → personalized dashboard | Pass — completes in well under 60s |
| Jaap counter persistence | Pass — count survived a full page reload |
| Library reading mode | Pass — meaning toggle, font resize, mark-as-read all work |
| Ask Guruji safety | Pass — medical question refused; devotional question answered; limit decrements |
| Sankalp create + Day 1 | Pass — journey created, Day 1 completed, Day 2 unlocked, streak earned |
| Auth protection | Pass — after logout, visiting `/home` redirects to `/login` |

Edge cases covered include empty/invalid inputs, special characters and Unicode
(Hindi/Devanagari renders correctly), refresh/back behaviour, the daily rate
limit, and `prefers-reduced-motion`. Full details are in `TEST_REPORT.md`.

Quality gates: `npm run lint` → 0 problems, `npm run build` → 0 errors, 15 routes
compiled.

4. Daily reminder notifications.

## What this project demonstrates

- Building a complete, multi-feature product end-to-end.
- Making pragmatic engineering tradeoffs and being able to justify each one.
- Writing safe-by-design logic (the Guruji classifier).
- Keeping code small, typed and readable so it can be defended in review.
